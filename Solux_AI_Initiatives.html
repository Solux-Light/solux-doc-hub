CRM Edits Needed

CRM EDIT New Fields

Add to the Lead object: lead_source (dropdown: Website / JivoChat / LinkedIn / Google Ads / Waalaxy / Trade Show / Email / Other), lead_type (New Project / Distributor / After-Sales / Study Request), country, urgency_score (1–5, set by AI), assigned_to (sales rep), first_response_sent (boolean + timestamp).

CRM EDIT Pipeline Rule

Leads cannot advance to "Qualification" stage unless lead_source, lead_type, and country are filled. This stops incomplete data reaching sales.

Automation — Lead Capture

INTEGRATE Joomla Contact Form

Add a webhook to the Joomla contact form → on every submission, POST the data to an n8n/Make.com workflow → auto-create CRM Lead record. Map the existing form categories directly to lead_type.

INTEGRATE JivoChat

JivoChat has a native webhook. On every new conversation, fire the webhook → n8n creates CRM lead with source = "JivoChat" and chat transcript attached.

INTEGRATE Waalaxy / LinkedIn Ads

Both have CSV export or Zapier connectors. Set a scheduled job (daily 8am) to pull new contacts from Waalaxy → create CRM leads, deduplicated by email.

AI Layer — The SDR Logic

AI LAYER Lead Classifier

When a new CRM lead is created, trigger an API call to GPT-4o mini. Send: inquiry text + country + lead source. Receive back: lead_type, product_family, urgency_score (1–5), suggested_assignee (by region/office). Write results back to CRM automatically. Cost: ~$0.001 per lead.

AI LAYER First Reply Drafter

After classification, AI drafts a personalised first response email in the correct language (French / English / Arabic based on country). Sales rep reviews and sends in one click — no writing from scratch.

AI LAYER Daily Owner Summary

Every morning at 8am, a scheduled job queries the CRM and sends the owner a summary email: leads received yesterday, classified breakdown by type, any urgent leads not yet assigned.

💡 Claude's Implementation Suggestions

The 3 Gaps to Fix

Gap 1 — Manual trigger

Every search requires a human to initiate it through Claude. Needs to be scheduled.

Gap 2 — Simple scoring

Claude reads and classifies but the scoring logic is basic. Needs structured criteria.

Gap 3 — No CRM link

High-value tenders don't automatically become Opportunities in the CRM.

Fix 1 — Schedule the Search

AUTOMATE Daily Cron Job

Remove the manual trigger. Set up a cron job (server) or n8n scheduled workflow to run the J360 fetch every weekday at 7am. It runs without any human action. Results are stored and scored automatically before the team arrives at work.

Fix 2 — Structured Scoring Criteria

AI LAYER Scoring Prompt Engineering

Rewrite the Claude prompt to return a structured JSON score for each tender:

  • product_fit (0–10): Does this tender mention solar lighting, street lighting, or outdoor lighting?
  • geography_fit (0–10): Is the country/region where Solux has an office or distributor?
  • scale_fit (0–10): Is the project scale appropriate (not too small, not too large)?
  • eligibility (Yes/No/Unclear): Does Solux meet stated certifications or origin requirements?
  • deadline_days: Days until submission deadline
  • recommended_action: "Pursue" / "Monitor" / "Skip"

Total score determines CRM action automatically.

Fix 3 — CRM Integration

CRM EDIT New Tender Object

Add a "Tender" object to the CRM with fields: tender_name, source (J360), country, deadline, estimated_value, score, status (New / Under Review / Submitted / Won / Lost), assigned_to.

AUTOMATE Auto-Create CRM Opportunity

When a tender scores ≥ 7/10 → the tool automatically creates a CRM Tender record and notifies the relevant regional sales manager via email. Score <7 → logged but not escalated. Sales manager reviews in the CRM, not inside Claude.

💡 Claude's Implementation Suggestions

Ownership Correction — Confirmed by Andrew & Phi, 3 Jul 2026

The 7-step list at left is Mehdi's own framing, but the engineers themselves confirmed a narrower scope: Engineering's work ends at Battery Sizing. Product Selection and Mechanical Study are Sales-owned, not Engineering — see the full 8-step ownership table in Business Analysis §2.3. That means "the engineer transfers results manually into the study document" (step 7 here) is really a hand-off to Sales, not an Engineering task to automate.

The Core Problem to Solve

Steps 4–6 in the workflow above are pure data re-entry. The engineer already has everything needed from the Phase 1 (Lighting Design) output — the system should just call PVGIS automatically and fill in the results, then hand the sizing output to Sales for product selection.

DEPENDENCY Phase 1 Must Come First

Phase 2 automation cannot be built standalone — it plugs directly into Phase 1 above. The PVGIS API call is triggered from within the same platform after a lighting design is selected. This means: Phase 1 must be reviewed and stabilised in Sprint 1 before any Phase 2 development begins in Sprint 3. The GPS coordinate field must also be added to the Phase 1 study request forms during Sprint 1 — without coordinates, there is nothing to send to PVGIS when automation is ready.

PVGIS API Integration

INTEGRATE PVGIS REST API

PVGIS has a free public REST API from the EU Joint Research Centre. Once the engineer selects a project location (GPS coordinates), the platform automatically sends: lat, lon, peak_power, system_loss, mountingplace → API returns monthly irradiance data, optimal tilt angle, peak sun hours by month. No manual web browser entry needed.

AUTOMATE Panel Sizing Engine

After PVGIS returns irradiance data, the platform automatically calculates panel size using Solux's 30% safety margin rule. Formula: panel_size = (daily_consumption / worst_month_sun_hours) × 1.30. Result auto-fills the study.

AUTOMATE Battery Sizing Engine

Platform automatically calculates battery capacity: battery_size = daily_consumption × 2 (minimum 2-day backup rule from solux.fr/en/study-lab). Result auto-fills. Engineer reviews — does not recalculate.

Dialux Export Parsing

INTEGRATE Parse Dialux Output

Dialux can export reports as PDF or XML. The platform should accept Dialux export upload → parse luminaire power, average illuminance, number of luminaires → auto-populate the study request form. This eliminates Step 2 manual data entry.

"Learn from Previous Successes"

AI LAYER Similar Project Recommender

When a new study request arrives (country + road type + luminaire count), query the archive of ~600 past studies → find the 3 most similar completed studies → suggest their panel and battery sizing as a starting reference. Engineer validates rather than calculates from zero.

PROJECT 3 — Product Knowledge Hub & Documentation Intelligence Platform
ROI Priority #4  ·  Sprint 2
📄 Mehdi's File — Verbatim

Vision

As Solux continues to expand its product portfolio, maintaining accurate technical information across all customer-facing documentation has become increasingly difficult.

Today, product specifications are maintained by the factory in a master Excel file, while Marketing manually recreates the same information across dozens of technical sheets, catalogues, brochures, presentations, websites and sales documents.

The objective of this project is not simply to digitize the existing Excel file.

The objective is to build a centralized Product Knowledge Hub that becomes the single source of truth for every technical specification used across the entire company.

This platform will serve as the foundation for Marketing, Sales, Engineering, Product Management and future AI assistants.

Current Situation — Product Information

Today the factory maintains one or several master Excel files containing official specifications for every product.

These files include information such as:

  • Product names
  • Product families
  • LED power

  • Dimensions
  • Weight

Strategy — Build the Intelligence Layer, Don't Replace the Tools

Mehdi's exact instruction: NOT to replace Mailchimp or Brevo

Both tools stay. The work is to build a middle layer that reads their data, surfaces insights, triggers automations, and syncs contacts with the CRM.

Step 1 — Connect to Mailchimp + Brevo APIs

INTEGRATE Pull Campaign Data

Mailchimp API + Brevo API both allow full campaign history export: open rates, click rates, unsubscribes, bounces — by campaign, by audience segment, by country. Pull this data weekly into a central analytics database. This is what makes the "easily answer questions" goal possible.

AUTOMATE Data Cleaning — Remove Invalid

Schedule a weekly job: query Mailchimp + Brevo for bounced, unsubscribed, or invalid emails → remove from active lists → flag in CRM so sales reps don't try to reach them by email. Keeps list quality high.

Step 2 — AI Campaign Intelligence

AI LAYER Campaign Performance Analysis

Monthly: GPT-4o mini reads all campaign performance data → generates a report: which product topics got the highest click rates by region, which subject lines worked in Africa vs France, which campaigns "stopped halfway" and why. Actionable recommendations for next month.

AI LAYER Audience Segmentation

AI analyses Mailchimp audience tags + click history → suggests better segmentation: e.g., "contacts who clicked Pro Line content 3+ times but have no CRM deal → flag as hot leads for sales." Bridges marketing engagement with sales follow-up.

Step 3 — CRM Loop (the Missing Link)

CRM EDIT Add Email Engagement Fields

Add to CRM Contact: last_email_opened, last_clicked_product, email_engagement_score (auto-updated from Mailchimp/Brevo API). When a contact clicks a product link → CRM automatically tags their interest. Sales sees this in the contact record.

AUTOMATE Hot Lead Alert

When a Mailchimp/Brevo contact opens 3+ emails AND clicks a product → automatically create a CRM task: "[Contact Name] has engaged with Pro Line content 3 times — follow up." Sales gets notified before they go cold.

Drip Sequence Automation

AUTOMATE Behaviour-based Sequences

Set up triggers in Mailchimp/Brevo: contact submits website form for Study Request → automatically enrolled in a "Study Request nurture" sequence (3 emails over 10 days: what is a lighting study / how to prepare / example project).
